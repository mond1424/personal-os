# Personal Agent (Personal OS) — 통합 설계 v0.9

> 캘린더 + 일기 앱 ↔ AI.
> 상태: **구현 1 완료 · 구현 2 부분 가동 · 원격 배포됨** (v0.9, 2026-07-21).
> 구현 1(AI 없는 뼈대) 전 화면 연결 완료, 구현 2 중 analysis 2-pass·Feelings 분류 동작.

## 존재 이유 — 실제 실패 사례 #1

> (2026-07) Personal OS를 설계하던 중, 다음 날 자격증 시험이 있음을 알고 있었음에도
> 새로운 아이디어에 대한 몰입이 우선되어 밤샘을 선택했고, 결국 시험에 응시하지 못했다.

- 실패의 원인은 **기억이 아니라 의사결정**이었다. 일정도, 알람도, 중요성 인식도 전부 있었다. 단기 몰입이 장기 목표를 압도했을 뿐이다.
- 따라서 본 시스템은 일정 기록·관리 도구가 아니라, **현재의 행동이 미래의 목표와 충돌하는지 판단하고 필요 시 개입하는 의사결정 보조 에이전트**를 목표로 한다.
- 이후의 실패 사례도 번호를 붙여 축적한다. 실패 사례는 요구사항의 원천이다. (이 축적의 자동화 계층이 6.5 Guard Memory 다.)

## 0. 핵심 원칙

1. **설계는 구현보다 오래 살아야 한다.** 이 문서는 구현 방법이 아니라 시스템의 철학과 구조를 정의한다. 구현 세부사항(UI, 알고리즘, 데이터 형식, 모델 선택)은 변경될 수 있지만, 여기에서 정의한 원칙과 엔티티 관계는 구현 방식과 독립적으로 유지되어야 한다.
2. **대화 컨텍스트는 저장소가 아니다.** 모든 기록은 구조화된 데이터로 외부에 저장하고, AI 호출 시마다 필요한 부분만 조립해 주입한다.
3. **저장은 논리 개체(entity) 기준, 캘린더는 뷰(view)다.** 저장 구조를 UI에 맞추지 않는다.
4. **원본과 파생 데이터를 명확히 구분한다.** 원본(tasks, periods, Feelings, Log, memo, self-Daily Score, Me, Guard 이력)은 사용자가 입력하거나 시스템 이벤트로 발생하는 사실 기록이며, 파생 데이터(Todo·Done·Missed 뷰, summary, 통계, 달성률)는 원본으로부터 계산·조립된다. 동일한 정보를 두 곳에 중복 저장하지 않으며, **물화(파생의 저장)는 cache 계층(summary/)에서만 허용된다.**
5. **3중 시간 스케일의 맥락 구조.** 장기: Me / 중기: 기간·목표 / 단기: Daily.
6. **summary ≠ analysis.** summary = cache(재생성 가능한 압축본), analysis = AI의 사고 결과(요청 시 생성, 영구 보존).
7. **시간 계산의 기준은 벽시계가 아니라 귀속일이다.** 하루 경계(05:00)를 쓰는 이상, 날짜가 관련된 모든 파생값(대기 일수·완료일·자동 마감·연장 기한)은 벽시계 날짜가 아니라 귀속일로 계산해야 한다. 두 기준을 섞으면 새벽 시간대에만 어긋나는 버그가 생기고, 그런 버그는 재현이 어렵다. (구현에서 실제로 발생 — 대기 일수가 SQL·앱에서 서로 다르게 나왔다.)
8. **기억 보조 < 의사결정 보조.** 차별점은 "잘 기억하는 비서"가 아니라 "장기 목표를 단기 충동으로부터 보호하는 개입"이다 (6장).

## 1. 데이터 계층

### 1.1 저장 구조 (entity-first)

```
data/
├─ daily/      YYYY-MM-DD.md — Diary (1.2)
├─ tasks/      (task_id).md — identity 원본 · 해야 할 일 (1.4)
├─ events/     (event_id).md — 일정 · 그 시각에 일어나는 사건 (1.7)
├─ periods/    (period_id).md — name, start, end, color, created_at, goals (2장)
├─ me/         장기 맥락 — 필드 단위 원본 + 변경 이력 (3장)
├─ guard/      개입 이벤트 로그 (6.5)
├─ summary/    daily / weekly / monthly cache (4장)
└─ analysis/   Analysis_YYYYMMDD (5장)
```

- **캘린더 = 조인 뷰.** 날짜 칸 = daily[그날] + tasks[그날 예정] + events[그날] + periods[그날 포함]. UI가 바뀌어도 저장 구조는 불변.
- **id/표시명 분리 (전 엔티티 공통)**: 파일명은 불변 id(생성시각 기반, 정렬 가능 — 예: `20260718-001`), 표시명(title)은 파일 내 메타. title 은 자유롭게 변경 가능하며 참조는 id 로 유지된다.
- 구현 시 D1 테이블에 1:1 매핑.

### 1.2 Diary (`daily/YYYY-MM-DD.md`)

**정체성: Diary 는 기록(record)이다.**
- tasks·periods·Me 는 **상태(state)** — 계속 변하는 현재.
- Diary 는 **하루 단위의 불변 스냅샷** — 원본 입력(Log·Feelings·memo)의 그릇이자, 시스템의 시간축. 상태는 이후에 변해도 그날의 측정 기록은 변하지 않는다.

| 섹션 | 성격 | 설명 |
|------|------|------|
| Todo | 파생 (뷰, **저장 안 함**) | tasks 중 그날 예정된 것 |
| Done | 파생 (뷰, **저장 안 함**) | 그중 status: finished |
| Missed | 파생 (뷰, **저장 안 함**) | 그중 status: not finished |
| Feelings | 원본 입력 | 눈금 or manual 서술 → AI 분류 (1.5) |
| Log | 원본 입력 | 타임스탬프 스트림 (아래) |

- 파생 3섹션은 읽을 때마다 tasks 에서 조립된다 (원칙 4). 과거 날짜에 대해서도 안전한 이유: tasks 의 과거 schedule 항목이 불변이기 때문 (1.3).
- **Missed 는 마감 시 확정되는 파생이다.** 진행 중인 당일 화면에는 존재하지 않는다 — 마감 전의 미완료는 그냥 Todo.
- **재배정 대기 = 최근 예정일이 오늘 이전이며 아직 완료되지 않은 task** (예정 이력이 있는 것에 한함 — 없으면 '대기'). 직전 마감분만이 아니라 지나친 모든 미완료가 잡힌다 — 하루 지나친 task가 어느 뷰에도 안 잡히는 잊힘을 막기 위해. Today 에 노출되어 미루기(재배정)를 유도한다.
- **하루 경계 = 05:00 (설정에서 변경 가능).** 경계 이전의 새벽 기록은 전날에 귀속된다. 경계 도달 시 자동 마감 + 이후 사용자에게 memo 작성 제안.
- **사람이 쓰는 것은 Log 와 Feelings 뿐.** 일기 마찰 최소화, 토큰 소모 0.
- **Log**: (timestamp, text) 수시 추가. 타임스탬프 자동(수정 가능), append-only.
  ```
  14:00  양자역학 공부하다가 Fourier에서 막힘.
  22:00  Personal OS 설계 시작.
  01:40  내일 시험인데도 계속 만들고 싶다는 생각이 듦.
  ```
  - 목적: 결과(Done)가 아닌 **과정**의 보존 — "새 개념을 만나면 앞으로 못 나가는 패턴" 류의 분석은 여기서만 가능하다.
  - 부산물: 기록 시각 = 활동 신호. 새벽 기록의 존재 = 각성 상태, 첫/마지막 기록 시각 = 수면 리듬 근사치 (6.6 의 입력).
- 모든 섹션 필수 아님. 일기를 건너뛴 날은 이후 "최소한 기분이라도" 유도.
- **하루 마감** = Feelings 확정 + Log 봉인. 마감 시점의 파생 섹션을 물화한 것이 기계적 daily summary 로 summary/ 에 저장된다 (4장). 마감 후엔 memo 추가만 가능. 미마감 시 경계 시각(05:00)에 자동 마감된다.
- hashtags: 검색 용도 우선. / 간략히 모드: Feelings 만 찍고 마감하는 최소 플로우.

### 1.3 불변성 규칙 (일반화)

> **과거에 귀속된 기록은 수정할 수 없다. 추가만 가능하다.**

- 대상: 확정된 daily 파일, tasks 의 과거 날짜 schedule 항목, **마감된 날의 events**, guard 이벤트 로그.
- **계획은 지울 수 있고, 기록은 지울 수 없다.** task 취소(삭제)는 마감된 날의 항목이 하나도 없을 때만 허용된다. 지나간 날에 걸쳐 있는 일은 취소가 아니라 완료·미룸으로 남긴다.
- daily 에는 memo 추가 가능(작성 시각과 함께). memo 추가 시 해당 날짜를 포함하는 summary 들은 stale 처리 → lazy 재생성 (4장).

### 1.4 tasks — identity 원본

```
tasks/20260701-003.md
  title: 퀴즈복습        # 표시명, 변경 가능
  period: 20260713-001   # 소속 기간 — 명시 (선택)
  created: 2026-07-01
  status: not finished
  schedule:
    - { date: 0701, 완료율: 80%, defer → 0704 }
    - { date: 0704, ... }
```

- **소속 기간은 명시 필드다 (period).** 날짜 조인으로 추정하지 않는다 — 기간이 겹치는 구간에서는 task 가 어느 기간 소속인지 날짜만으로 판정할 수 없다. Works 기간별 세그먼트·기간색 노치는 이 필드를 전제로 한다. 무소속 task 허용, 대기(미배정) task 도 기간에 속할 수 있다.
- **미루기 = schedule 배열에 항목 추가.** 복사 없음, identity 는 구조적으로 보존.
- **미루기의 새 예정은 완료율 0에서 시작한다.** 미루기는 기존 항목의 수정이 아니라 새 예정의 생성이다. 이전 항목의 완료율(예: 80%)은 "그 예정일까지 얼마나 진행했는가"라는 과거 기록이지 새 예정일의 현재 진행률이 아니다 — 이어받으면 새 일정이 이미 진행된 것처럼 보이고 예정일별 진행 기록이 섞인다. 과거 진행률은 이전 항목에 영구 보존하고, 새 항목은 새로운 시작점으로 취급한다.
- 동명 task 허용: 구별은 id 가 한다.
- **이월 횟수는 파생값이다** (schedule 배열에서 계산) — 데이터에도 화면에도 기호 표기는 없다. UI 는 **경고 배지(빨간 원 + 흰 느낌표) + "n회 이월" 텍스트**로 표시한다. (v0.6 의 `@` 표시 규칙과 title `@` 입력 금지 규칙은 폐기)
- "이 과제 3번째 미루는 중"은 패턴 분석의 1급 신호. 완료 O / 미완료 X → 완료율 다이얼 → 미루기 날짜 선택(캘린더에서 직접 탭) 플로우 동일.
- **2주 제한은 미루기에만 적용된다 (확정).** 규칙의 목적은 "미루기가 무기한 연기가 되는 것"을 막는 것이므로, 미루기·재배정의 대상 날짜만 오늘~+14일로 제한한다. **신규 일정 지정(빠른 추가·대기 확정)에는 상한이 없다** — 시험처럼 한 달 뒤로 확정된 일도 넣을 수 있어야 한다. 지난 날짜만 거부한다.
- **재배정(마감된 날에서의 미루기)은 insert-only 다 (확정).** 원 항목은 Missed 로 동결된 채 남고(그 날 하지 못했다는 사실은 기록이다), 새 예정만 추가된다. 응답은 일반 미루기와 구분해 `reassigned` 로 표시한다.
- **완료는 '살아 있는 항목'에 100% 를 붙인다 (확정).** 완료를 누른 날(finished_on = 그 순간의 귀속일)과 그 일의 예정일은 다를 수 있다. 오늘 날짜의 항목만 갱신하면 "완료인데 진행률 0%"라는 모순이 남으므로, 미뤄지지 않은 마지막 항목의 완료율을 100 으로 만든다. 단 그 날이 이미 마감됐다면 건드리지 않는다(1.3). 화면은 예정일과 완료일이 다르면 둘 다 보여 준다.
- **`schedule: []` 허용 = 미배정 상태, UI명 '대기'.** 날짜를 지정하면 첫 schedule 항목이 생긴다. Today·캘린더는 schedule 조인이므로 미배정 task 는 뷰에 잡히지 않는다 — 가짜 날짜 강제는 Todo·Missed·패턴 신호를 오염시키므로 금지.
  - **잊힘 방지**: Today 에 개수 + 최장 체류 일수를 상시 노출. 최대 체류 **21일** — 다가갈수록 일수 표시가 탁한 적색 계열로 점진 변화(원색 금지). 초과 시 앱 실행 시 **차단 팝업** → '지금 정하기' → 해당 task 상세로 직행, [일정 확정 / 대기 연장 / 완료] 중 택일 전까지 진행 불가.
  - **연장 = 대기 앵커(wait_anchor)를 연장한 현재 시각으로 갱신.** 기한 = 앵커 + 21일, 반복 가능. 연장 이력은 추가 전용(append-only)으로 보존한다 — 연장 횟수·간격이 이후 패턴 신호가 되며, 활용 수위는 9장 #3(대기 재노출 강도)과 함께 정한다.
- **반복(recur) ≠ 미루기(defer).** 미루기 = 하나의 약속이 identity 를 유지한 채 이동. 반복 = 규칙이 매 주기 **새 약속(독립 task, 동명·id 상이)** 을 생성 — 회차끼리 별개라 이월 신호가 섞이지 않는다 ("동명 task 허용" 원칙이 이 구조를 지지). 규칙 문법·선생성 범위는 미확정(9장).

### 1.5 feelings 입력

- 토글: 체크리스트 모드(점수 직접 입력) / manual 모드(자유 서술 → AI 분류).
- 세부 필드: energy / stress / focus … (settings에서 구성). 분류 결과는 DATA 축적 → 그래프.
- **필드는 시계열의 축이다 — 추가는 쉽게, 제거는 어렵게.** 축을 바꾸는 것은 데이터를 바꾸는 일이다. 사람마다 중요한 축이 다르므로(수면·불안·통증) 고정하지 않되, 기본 3개(energy·stress·focus)를 강하게 밀고, 기록이 있는 축을 뺄 때는 축적된 일수를 보여 주며 경고한다. 지난 기록은 남지만 그 지점부터 끊겨 비교가 어려워진다는 사실을 명시한다. 매일 매기는 부담 때문에 동시 축은 소수(6개)로 제한한다.
- 척도: 1~10 (눈금 1 단위), 직접 타이핑 시 소수점 허용. 입력 UI 는 채움형 슬라이더가 아니라 **심리검사식 눈금 수평선**(눈금 탭) — 채움 바가 만드는 '끝까지 채워야 할 것 같은' 편향 제거.
- manual 모드의 분류 결과는 **하루 마감 시 확정 → 원본 편입**.
- 분류 비용은 소형 모델 기준 무시 가능. 절약 시 analysis 요청 시점 일괄 분류 가능.

### 1.7 일정(event) — 할 일과 다른 것

> 캘린더에 적은 것이 자동으로 '해야 할 일'이 되면, 완료율·이월 신호가 오염된다.

| | task (할 일) | event (일정) |
|---|---|---|
| 정체 | 해야 할 일 — 내가 수행하는 것 | 그 시각에 일어나는 사건 — 시험·약속·수업 |
| 가지는 것 | schedule 배열 · 완료율 · 이월 · 대기 | 날짜 · 시각(선택) · 기간 소속(선택) |
| 없는 것 | — | 완료·미루기·이월 개념이 없다 |
| 다루는 곳 | Works · Today | **캘린더에서만** 추가·삭제 |
| 신호 | 미룬 횟수 = 회피 신호 | 사건의 존재 자체 = 맥락(Guard 판단 입력) |

- 일정은 Todo·Missed·달성률 계산에 들어가지 않는다. 반대로 Guard 에게는 1급 입력이다("내일 10시에 시험이 있다").
- **마감된 날의 일정은 수정·삭제할 수 없다** (1.3). 그 날 무슨 일이 있었는지는 기록이기 때문이다.
- 시각은 선택이다(없으면 '종일'). 시각이 있으면 캘린더 셀과 팝업에서 앞에 붙는다.

### 1.6 self-Daily Score

- 하루에 대한 **주관적** 종합 점수. 계산하지 않고 독립 입력 유지.
- 목적: feelings·tasks·기간과 교차해 "이 날은 왜 이랬지?" 분석의 축.
- 입력 UI: **막대그래프** — 가로축 날짜(최근 2주, 가로 스크롤), 세로축 점수. 과거 막대는 연하게, 오늘 막대는 진하게 — 전날들과 비교하며 오늘 칸을 탭·드래그해 높이로 매긴다. 안내 문구는 "오늘에 점수를 매긴다면" (계산하지 않음을 굳이 명시하지 않는다).

## 2. 기간과 목표

### 2.1 데이터 (`periods/`)

- 기간 = { name, start, end, color, **created_at**, goals } — 파일명은 id (1.1 공통 규칙).
- **기간 ⊇ 목표**: 경과 일수(2/5 Days)와 달성률(31%) 표시.
- **달성률 (확정) = 기간 내 task들의 완료율 다이얼 값 평균.** 완료 = 100, 미착수 = 0, 부분완료 = 다이얼 %. (파생 데이터 — 저장하지 않고 계산)

### 2.2 캘린더 렌더링

- **기간은 배경이다.** 형광펜처럼 내용을 덮는 것이 아니라 뒤에서 구간을 물들인다. 밴드는 셀 아래 계층(z-index 0)에 깔리고, 날짜·일정·할 일 글줄이 그 위에 온다. 글줄에 흰 배경을 두지 않는다 — 기간색 위에 흰 조각이 뜨면 배경이 아니라 스티커처럼 보인다.
- **겹침 = 경계선 모델 (v0.9 재정의).** 임의 구간에서 활성 기간이 n개면 밴드를 n등분하고 created_at 순으로 위→아래 배정한다. 규칙은 셋:
  1. **밴드는 셀 높이를 꽉 채운다.** 위아래 여백을 두지 않는다.
  2. **전환은 반 칸 폭의 S-곡선, 그 밖은 완전 수평.** 배치가 달라지는 지점에서만 곡선으로 이동한다.
  3. **경계는 공유한다.** 새로 시작하거나 끝나는 기간은 "자기가 끼어들 자리의 경계"로 수렴하며, 이웃 기간의 전환 곡선과 **같은 구간·같은 곡선**을 쓴다. 두 밴드가 한 선을 나눠 갖기 때문에 교차도, 사이의 흰 틈도 생기지 않는다.
- **시작·끝에 이웃이 없으면 곡선 없이 수직으로 자른다.** 기간이 혼자인 구간은 칸을 그대로 다 채운다 — v0.7 의 "두께 0에서 열고 닫는" 렌즈 모양은 폐기한다(단독 기간이 시작·끝에서 얇아져 배경 역할을 못 했다).
- **셀 표시**: 구석의 작은 날짜 숫자 + **일정**(굵게, 시각이 앞에) + **할 일** 글줄(기간색 노치) + 일기 존재 밑줄 마커. 2줄 + "+n" 오버플로. 완료된 항목은 옅게 + 취소선, 미룬 항목은 옮겨간 날짜에만 표시(지난 날에는 취소선으로 남긴다). 상세는 전부 셀 탭 → 날짜 팝업.
- **날짜 팝업 = 조인 조립 + 편집 지점.** 기간 칩 + D-day + **일정 섹션**(추가·삭제) + **할 일 섹션**(추가, 줄을 누르면 task 상세) + 보호 규칙 + (과거) 일기·memo 를 원본에서 계산해 조립한다. 오늘·앞으로의 날짜는 여기서 바로 편집한다 — 일정을 고치려고 다른 탭으로 이동하지 않는다. 과거 = 읽기 + memo 추가만.
- 구현: SVG path — 경계선들을 계산한 뒤 사이를 채움. 주(행) 단위 독립 렌더. 달 넘김은 좌우 버튼과 **가로 드래그** 둘 다. 가로 화면의 사이드 패널은 가로 레이아웃 설계 시 확정.

## 3. Me 페이지

- 내용: 방향(장기 목표), 관심사, 진로, 성격, 생활패턴 — 모든 analysis 의 장기 맥락 프레임.
- **목표는 이원 표시**: ① **방향** — 시한 없는 장기 서술, Me 의 원본 필드. ② **지금** — 활성 기간(periods)의 목표를 조인한 **파생 표시, 비저장** (원칙 4: 중복 저장 금지). 중기 목표의 진실은 periods 에 있으므로 Me 오염 면적도 장기 서술로 좁아진다.
- 갱신: 초기 사용자 기입 → 이후 **AI 제안(diff) → 사용자 승인.** 승인 화면 경고: "AI의 판단이 급진적일 수 있고, 자기 서술은 제안에 흔들리기 쉽다."
- **변경 이력 보존 + 이력 자체를 데이터로 활용** — Me 의 변화 궤적이 분석 입력이 된다.
- Me 오염 = 이후 분석 전체의 오염. 신중한 갱신 절차 유지.

## 4. Summary 파이프라인 (cache 계층)

```
원본 (daily, tasks) ──(하루 마감)──▶ daily summary ──▶ weekly ──▶ monthly
```

- 성격: cache. **파생의 물화가 허용되는 유일한 계층** (원칙 4). 언제든 재생성 가능, 자체 보존 가치 없음.
- daily summary 두 층위:
  - (a) 기계적 층위 = 마감 시점 파생 섹션 + 필드값의 물화 (1.2)
  - (b) AI 압축 = 서술 요약 (Log 포함)
  - ※ "AI 압축은 주 단위로만" 안은 기각: 고정 주 경계가 유기적 구간에 단절을 만듦.
- weekly / monthly 는 AI 압축 롤업.
- 무효화: memo 추가 시 해당 날짜 포함 daily → weekly → monthly stale 마킹. 재생성은 lazy.

## 5. Analysis 파이프라인 (사고 계층)

### 5.1 트리거
- 자동 생성 없음. **사용자 요청 시에만** 생성 (토큰 통제).
- **투명성**: 조립된 컨텍스트 원문을 사용자가 그대로 열람할 수 있어야 한다. 무엇이 모델에 들어가는지 보이지 않으면 토큰 통제도, 결과에 대한 신뢰도 성립하지 않는다.

### 5.2 입력 조립
```
[ Me ] + [ 요약/원본 윈도우 ] + [ Today ] + [ 사용자 prompt ]
```
- 종합 맥락 원칙: 현재 기간, 앞으로 올 일, 지나간 일을 함께 주입.
- **윈도우 규칙 (확정):** 이번 주 경과 ≥ 4일 → 이번 주 raw만 (4~7일) / ≤ 3일 → 지난주 raw 포함 (8~10일) / + 그 앞 완결 주 weekly summary (7일) ⇒ 총 **11~17일**.
- 장기 방향: AI가 맥락 보고 윈도우 동적 판단.

### 5.3 2-pass 생성
1. **1차**: 과거 analysis 없이 독립 생성 (앵커링 방지).
2. **2차**: 1차 + 과거 analysis 를 읽으며 **추가** 작성. 1차 수정 금지 (독립 시선 + 연속성).

### 5.4 저장
- `Analysis_YYYYMMDD = { prompt, context_meta, result: [1차, 2차] }` — 영구 보존. context_meta 는 조립된 컨텍스트 윈도우의 기록(어떤 raw·summary 가 주입됐는지) — 재현·감사용.
- **표시**: 기본은 1·2차를 **통합한 산문**으로 보여준다 (분리 열람은 원문 보기). 저장의 [1차, 2차] 분리는 유지 — 앵커링 방지는 데이터 층의 속성이다. 목록 UI 는 접이식 카드(제목·날짜 헤더 탭으로 개폐), 3건 초과 시 '더보기' → 게시판형 전체 목록(내비 바 제외 전체 화면).

## 6. 개입 메커니즘 (Guard)

> 실패 사례 #1 에서 도출. 기존 일정·일기 앱과의 최대 차별점.

### 6.1 개입 단계

| Lv | 이름 | 예시 |
|----|------|------|
| 1 | 알림 | "내일 오전 10시에 시험이 있습니다." |
| 2 | 맥락 경고 | "지금 새 프로젝트를 시작하려 합니다. 내일은 시험이 예정되어 있습니다." |
| 3 | 위험 판단 | "현재 01:30. 지금 계속하면 예상 수면이 4시간 이하 — 응시 실패 확률이 높습니다." |
| 4 | 적극적 개입 | "이 작업은 저장해 두겠습니다. 시험 후 이어서 진행을 권장합니다." |

- 판단에 쓰이는 입력은 6.6 참조.

### 6.2 핵심 원칙: 사전 서약 (pre-commitment)

- Level 1~4 는 '그 순간의 사용자'를 설득하는 구조다. 그러나 개입이 필요한 시점(새벽, 몰입 중)은 설득이 가장 통하지 않는 시점 — 실패 사례 #1 에서 사용자는 알고 있었고, 이미 결정한 상태였다.
- 따라서 결정 시점을 옮긴다: **'미리 정한 나'가 규칙을 만들고, 집행은 자동으로.**
- 구현: **기간/일정에 보호 규칙 부착.** 예) 시험 등록 시 → 전날 00:00 이후 보호 모드(신규 작업 차단, Level 3~4 활성).

### 6.3 Override 는 금지가 아니라 마찰

- 절대 잠금은 도구 이탈로 귀결. 해제는 가능하되 비용 부과: 사유 한 문장 타이핑 + n분 대기.
- 해제 사유·시각은 guard/ 에 기록되어 6.5 의 데이터가 된다.

### 6.4 강제력의 현실적 한계

- **규칙 평가 = 서버.** Cloudflare Worker Cron 이 주기적으로 규칙을 평가하고 Web Push 로 도달한다 — 앱이 닫혀 있어도 개입 가능.
- PWA는 다른 앱 종료·PC 전원 제어 불가. 소프트웨어 개입 상한 = 앱 내 마찰 + 알림.
- **환경 설계 병행**: 오전 시험 회피(오후 접수), 전날 PC 예약 종료, 미션 알람 — 시스템 밖 장치가 최종 방어선.

### 6.5 Guard Memory — 개입의 경험 축적

Guard 는 규칙만 적용하는 시스템이 아니라, **과거 개입의 결과를 기억하는 시스템**이다.

- Guard 가 발동하거나 Override 가 이루어질 때마다 하나의 **Guard Event** 를 기록한다.
- Event 는 최소한 다음 정보를 가진다:
  - 발동 시각 / 발동 원인(규칙) / Guard Level / 사용자 반응(수용·Override) / **Override 사유(6.3 에서 타이핑한 문장)** / 연결된 task 또는 period / 최종 outcome(성공·실패)
- **outcome 은 Guard 가 직접 판단하지 않는다.** 이후 해당 task 또는 period 의 실제 결과와 연결되어 사후 확정된다.

**활용** — Guard Memory 는 이후 개입의 입력이 된다.
> "지난 컴활 시험에서도 같은 사유로 Override 를 선택했고, 결국 시험에 응시하지 못했습니다."

일반적인 조언보다 **개인의 실제 전례가 훨씬 강한 설득력**을 갖기 때문이다.

**자기 보정(Self-correction)** — Guard 는 일방적으로 강해지는 시스템이 아니다.
- 특정 규칙에서 Override 이후에도 **성공이 반복**된다면 → 그 규칙은 과도하게 보수적일 가능성.
- Guard 를 무시한 뒤 **실패가 반복**된다면 → 그 규칙은 강화되어야 할 가능성.
- 즉 Guard 는 **'실패를 줄이는 방향으로 스스로 보정되는 시스템'** 을 목표로 한다.

### 6.6 Guard 판단 입력

Guard 는 일정만 보고 판단하지 않는다. 개입 여부는 여러 종류의 정보를 종합하여 결정한다:

- 현재 시각 / 다음 일정까지 남은 시간 / 현재 기간 / 최근 Log 활동 / 최근 Feelings / self-Daily Score 추세 / 최근 수면 패턴(추정 — 1.2 의 Log 시각에서) / Guard Memory

구체적인 위험도 계산 방식은 구현 단계에서 결정한다.

### 6.7 개입의 목적

Guard 의 목적은 사용자의 행동을 강제로 통제하는 것이 아니다. 목적은 —

> **장기 목표와 단기 행동이 충돌하는 순간을 인식시키고,
> 사용자가 냉정한 상태에서 미리 정한 원칙을 실행하도록 돕는 것.**

따라서:
- 언제나 Override 는 가능하다.
- 다만 Override 에는 일정한 마찰을 둔다.
- 개입의 성공은 "실패를 0으로 만드는 것"이 아니라 **실패 확률을 낮추는 것**이다. 성공 지표는 실패 간격의 증가다.

| Today | 오늘의 기록·실행 (Diary 쓰기) | 헤더(기간 칩) → **오늘 일정**(읽기 전용, 관리는 캘린더) → Todo 조인 뷰(Done 기본 펼침, 재배정 대기 행) → 대기 상시 행 → Feelings(눈금/서술 · ⚙ 필드 고르기) → Log(입력줄 하단 고정) → Score · 마감 |
| Calendar | 거시 일정 + 일정 관리 + 과거 일기 읽기 | 월 그리드(배경 밴드 · 경계선 모델) ↔ 목록 토글 · 날짜 팝업(일정·할 일 추가/편집) · 기간 리스트(추가·편집·삭제) · 날짜 선택 모드 |
| Works | tasks 원본 관리 | 세그먼트: 예정 / 대기 / 이월 중 / 기간별 / 완료. task 상세 시트(title 수정 · schedule 이력 · 완료율 0~100 · 미루기 / 연장 / 완료 / 취소). FAB 빠른 추가 |
| Analysis | 사고 요청·열람 | 새 분석(prompt + 조립될 컨텍스트 미리보기) · 지난 분석 접이식 카드 · 더보기 게시판 |
| Me | 장기 맥락 + 설정 | 방향 · 지금(파생) · 관심사 · 생활 패턴 · 변경 이력 · ⚙ 설정 |

- **설정 구성**: 하루 경계 · Feelings 필드 · 테마 · 튜토리얼 / **AI 연결**(제공자·키) · 앱 접근 토큰 · 모델 Low · 모델 High / 표준시 오프셋 · 데이터 내보내기 · Guard(구현 3). 자주 만지는 것이 위, 한 번 정하고 마는 것이 아래. **자주 바꾸지 않는 값(토큰·키·오프셋)은 잠가 두고 [변경]을 눌러야 편집**되며 저장 시 재확인한다.
- **첫 실행 튜토리얼**: 화면 이름이 아니라 "왜 이 도구인가"부터 — 기록이 아니라 판단을 돕는 도구라는 것, Log 는 과정을 남긴다는 것, 마감은 봉인이라는 것, 미루기는 신호가 된다는 것. 설정에서 다시 볼 수 있다.
- **테마**: 자동(OS) / 라이트 / 다크. 색은 변수 한 벌 교체로 처리한다.
- **제스처**: 화면 가로 스와이프 = 탭 이동. 캘린더 그리드 위의 가로 드래그 = 달 넘기기(탭과 구분되도록 이동 임계값을 두고, 드래그 직후의 클릭은 삼킨다).
- **되돌릴 수 없는 동작 앞에는 확인을 둔다**: 하루 마감("오늘 하루를 마감할까요?" — 무엇이 잠기는지 설명), 일정 취소, 키 저장.
- **취소는 회피일 수 있다.** task 취소 문구는 '삭제'가 아니라 '취소'이며, 미룬 적이 있는 일은 미룬 횟수를 보여 주고 **[차라리 미루기]** 를 대안으로 제시한다. 2회 이상이면 확인 버튼이 "그래도 취소"로 바뀐다. 더 강한 마찰(사유 한 문장 + 대기 + 사유 축적)은 Guard 의 Override 구조(6.3·6.5)와 같은 것이므로 구현 3 에서 통합한다.
- **연결 실패는 화면을 깨뜨리지 않는다.** 데이터가 준비될 때까지 부팅 오버레이가 화면을 덮고, 실패하면 [다시 시도](401 이면 [토큰 입력])를 남긴다. 앱을 다시 열면 조용히 새로고침하고, 귀속일이 바뀌었으면 날짜를 다시 잡는다.
- Guard 는 탭이 아니라 **레이어**(푸시 · 배너 · 모달 · 전체 화면)로 모든 탭 위에 뜬다 — 구현 3.
- 가로(태블릿/PC) 레이아웃은 세로 화면 실사용 검증 후 설계.

## 8. 인프라 (구현 단계 메모)

- 프론트: PWA (홈 화면 추가) — 의존성 없는 정적 에셋(HTML·CSS·JS 3종)을 Worker 가 함께 서빙한다. 렌더링 계층(app.js)과 fetch 계층(api.js)을 분리한다.
- 호스팅: Cloudflare — Workers + D1 + [assets]. 카드 등록 불필요. 배포는 `wrangler deploy` 한 번, DB 는 건드리지 않는다(스키마 변경 시에만 migrations apply).
- 얇은 서버 = Worker 1개: AI 중계 + CRUD. 상시 서버 불필요.
- 1.1 구조 ↔ D1 테이블 1:1. 1.3 불변성은 API 가 아니라 **트리거로 DB 계층에서 강제**, 파생은 뷰·파라미터 쿼리, 회귀 검증은 스모크 테스트.
- **모델 이원화 (High-Low mix)** — 토큰 통제의 축.
  - `model_low`: 일상·잦은 호출 (Feelings 서술 분류)
  - `model_high`: 추론 (analysis 2-pass, 이후 Guard 판단)
  - 값은 `provider/model` 형식이라 Low·High 를 **서로 다른 회사 모델로** 지정할 수 있다.
- **AI 연결은 제공자별로 여러 곳을 등록해 둔다.** 키는 서버(설정)에 저장하되 조회 API 는 값을 돌려주지 않고 설정 여부만 알려준다. 앱에서 키를 넣을 수 있어야 배포 후 시크릿을 만지지 않는다. 서버 시크릿은 키가 없을 때의 폴백으로 남긴다.
  - **키 관련 오류는 원인을 말해 줘야 한다**: 어느 제공자·어느 모델에서 몇 번 오류가 났는지 그대로 보이고, 실제로 한 번 호출해 보는 **연결 테스트**를 제공한다. (401 의 실제 원인은 대개 ① 붙여넣기 과정의 개행·공백 ② 앱 접근 토큰과 AI 키의 혼동이었다 — 키는 항상 trim 하고, 두 입력을 물리적으로 분리한다.)
- **인증 두 겹**: 앱 접근 토큰(서버 접속, 기기 로컬 저장) ≠ AI 키(모델 호출, 서버 저장). 이름과 입력 위치를 분명히 갈라 놓는다.
